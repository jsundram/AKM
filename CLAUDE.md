## What this is

A pure-pull **PWA** that shows the day at the AKM Chamber Music Festival: Jason's
rehearsals (room + coach), an hourly weather curve, and a grace-note quote. It fetches the
live schedule and forecast **client-side** ‚Äî no backend, no scheduled job ‚Äî and caches the
whole festival week so it works offline once loaded. Hosted on **GitHub Pages**, installed to
the iPhone home screen. Opening it *is* the refresh. Three companion pages (also pure-pull,
offline, SW-precached), reached from a **shared top nav** (`nav.css`) on every page: a
participant **roster**, an offline village **map**, and a co-performance **network** graph.

## Architecture

Static files only. Nothing server-side.

- `index.html` ‚Äî app shell + all CSS (the design system). Mount point `#app`, a day-chip row, and an online/as-of status line.
- `nav.css` ‚Äî the **shared top-nav** (`.apphead`) used by every page: the **wordmark** (‚Üí `about.html`, collapses to "AKM" on narrow screens) plus a `nav` of page links (Today / Map / Roster / Network / Concerts) with the current page marked `aria-current`. Uses each page's own palette tokens so it themes (incl. dark) without redefining them; the markup is static per page, this file is the only styling. Replaced the old per-page footer links.
- `app.js` ‚Äî the brain: fetch ‚Üí parse ‚Üí render ‚Üí cache. Ported 1:1 from the earlier Python runner. Also reads the **shared roster** via `roster-data.js` (`Roster.cached()` / `Roster.pull()` in `refresh`) and derives each person's first name ‚Üí Notes-URL (`coachMap`), so a **coach's name in a rehearsal/lesson card links to their bio** (`coachLink`; first-name match ‚Äî faculty URL-holders don't collide; handles a compound `"Gijs/Nathan"` cell; underlined in muted ink, *not* the reserved brass). On a **self-added event** the `with` names each become a **WhatsApp deep-link** when the roster has that person's number (`waName`/`whoWa` resolve the stored `who` token ‚Üí `Roster.cached()` row ‚Üí `wa.me/<digits>`, reusing `whoRoster`'s first-name/collision token logic + the `WA_SVG` glyph; the link opens the chat with no pre-filled text, and a name with no number stays plain ‚Äî never a dead link). The card's delegated click handler already skips any `<a>`, so tapping a name opens WhatsApp instead of the editor. Each rehearsal/coaching/unscheduled card also carries its block's shared **Group letter** (A‚Ä¶E) as a large, faint Fraunces mark in the **bottom-right corner** (`gLetter`), tinted to the card's accent ‚Äî a coordination handle everyone in the block shares (players *and* coaches), deliberately an oversized corner label rather than a meta-row chip so it can't be misread as a room code (A1/A2). Threaded as a spare tuple index (`mineOf`/`freeOf`/`coachingOf`), so it never collides with `[6]` = composer key. Cards with a letter are stamped `.gcard` (reserves a piece/meta right gutter so nothing runs under the mark; the kicker/Ôºãadd row stays full-width). A **(KS) dress-rehearsal** card (`dressOf`, your own playing ‚Üí brass) carries a big brass **"!"** in the same corner slot instead (`gBang`/`.gbang`) ‚Äî the block is a short, fixed, easy-to-miss window, so the mark reads as urgency ("be on time"), not a room code or coordination handle. **Informational events** (an open talk, a discussion, a yoga session ‚Äî e.g. "Emi Presentation in AH", "Multi-Platform Careers Discussion ‚Ä¶ in AH", "Yoga Session with Elinor in AH") parse into `day.info` (`[start, end, title, room]`) and render for **everyone** regardless of identity as a neutral (never-brass) `.infocard` carrying a muted circled-**"i"** SVG corner mark (`INFO`/`.ginfo`) in the group-letter slot ‚Äî people have stopped double-checking the sheet, so these get surfaced rather than silently dropped. Detected in `parse` **by shape, not a keyword whitelist**: a prose cell (has lowercase) that names its room *inline* with a word before the "in ROOM" (`Emi Presentation in AH`) ‚Äî which a standalone "in A4" rehearsal note (nothing before "in"), a coach-tagged rehearsal cell (`- P`/`- C`), an all-caps banner, and any `faculty` item all fail ‚Äî so a rehearsal, a Faculty Dress, and a meal never masquerade as one. Works in both sheet layouts: time + title in one letter-spaced cell (Sat) or split across cells (Fri). Runs after meals/concerts/dress (they `continue` first) and consumes the row so it can't fall through into a junk rehearsal. A **faculty presenter's name in the title links to their bio** (`linkNames` ‚Äî any word matching a `coachMap`/`COACHES` key, i.e. a roster URL-holder, becomes the same muted bio link a coach's name gets): "Emi"/"Tanya" resolve (a first-name collision like the two Tanyas lands on the faculty URL-holder for free, since only faculty carry bio URLs), while a non-faculty presenter (Elinore, W1 ‚Äî no bio URL) stays plain text.
- `roster-data.js` ‚Äî **shared roster data layer** for the schedule, roster, and map pages. One gviz JSONP pull + one header-tolerant parse (gviz flip-flops between labelling the cols and dumping the header into row 0 ‚Äî try labels, else row 0) + one `localStorage` cache (`akm-roster`). Also owns the two cross-page identity bits: **`me()`/`setMe()`** (the picked identity, `akm-me`) and **`hotelPoi()`** (roster Hotel value ‚Üí map POI name; "Obernosterer Apartment" ‚Üí the "Haus Obernosterer" pin + the Kultursaal apartment ‚Äî "Obernosterer AirBNB" is its own POI at Liesing 50, so it self-resolves). Also owns **`instKind()`** ‚Äî the single instrument classifier (abbreviations *and* full names ‚Üí `v`/`va`/`vc`/`bass`/`piano`/`clarinet`/`oboe`/`v/va`/`""`, unknown never guessed): roster.html's `instLabel`, app.js's `fams` (lesson families), and network.js's Winds grouping all derive from it, so they can't drift (they used to be three prefix-ladders ‚Äî a "Cello" read as Clarinet on the roster because only that copy lacked the `ce` branch; `scripts/inst-test.js` pins it). network.html loads `roster-data.js` too for this. Exposes `window.Roster` (`SID`/`GID`/`KEY`/`parse`/`cached`/`pull`/`me`/`setMe`/`hotelPoi`/`instKind`); loaded **before** the page scripts. Single source of truth so the gviz-header quirk is fixed once; and since the schedule page (opened far more often) primes the same cache, the roster page renders offline even if it was never opened online.
- `ping.js` ‚Äî **usage pings**, loaded on every page. Each open (and throttled resume) appends `{ts, page, uid}` to a `localStorage` queue (`akm-pings`, capped) then flushes it via `no-cors` GET to a private Apps Script endpoint (`scripts/analytics.gs`, bound to a **private** analytics sheet in Jason's account ‚Äî one row per open; the sheet is the dashboard). Offline opens flush on the next online one. `uid` = first 8 hex of SHA-256 of the picked name ‚Äî no names in the stream; the sheet-side `=UID(name)` formula rebuilds the hash‚Üíname key from the roster anytime (stable across roster renumbering, unlike the `#` column). Empty `uid` = no identity picked ‚Äî the tripwire for strangers hitting the public URL. Reads `localStorage["akm-me"]` directly (no `Roster` dependency ‚Äî network.html doesn't load roster-data.js). The page name comes from the pathname, so a **directory URL would read as "index"** ‚Äî pages served at a `/dir/` URL (usage/) set `window.AKM_PAGE` before loading this. Beyond opens, it exposes **`window.AKMPing.event(action, toName)`** for **feature-use events**: concerts.html calls `event("kudos", <recipient>)` when a kudos chip is tapped, so Jason can tell whether applause is a couple of friends or widespread. An event row carries an **empty `page`** (so both usage crunches ‚Äî which count opens and filter on a non-empty page ‚Äî skip it automatically, no crunch/test change), the sender's uid in `who`, the event name in a new **`action`** column, and the **recipient's** uid in a new **`to`** column (`toName` hashed with the same recipe, so no names enter the stream and both ends reverse through `=UID()`); events are unthrottled (each is a real tap) and flush re-reads the queue each pass so an event enqueued mid-flush isn't clobbered. **`URL_` empty = endpoint not deployed: pings queue silently, nothing is sent, nothing errors** ‚Äî and the backlog flushes once the URL lands. about.html's privacy note deliberately doesn't mention pings yet (Jason wants the wording just right first).
- `sw.js` ‚Äî service worker; precaches the shell for offline, cache-first for Google Fonts (so the type survives offline), passes cross-origin *data* calls (gviz, open-meteo) through to network. Bump `V` when the shell changes ‚Äî a new `V` is what evicts the stale cache on activate, so a `SHELL`-file edit that forgets it ships to the repo but never to installed clients (this bit the v77 concert-card rewrite). A warn-only pre-commit lint (`scripts/sw-lint.py`) nags when a commit stages a precached `SHELL` file but leaves `V` unbumped. The **schedule footer prints the installed `V`** next to "switch user" (`verHtml`/`checkVer`/`.swver` ‚Äî read from `caches.keys()`, since `caches.open(V)` names the cache after `V`, so it can't drift from a hardcoded string): it shows the version *this device actually took*. `checkVer` (on boot + foreground) also fetches the deployed `sw.js` (`?_=`+no-store to dodge both caches) and compares ‚Äî the tag is **green (`.ok`) when current, red + underlined (`.behind`, "¬∑ akm-v87 ‚Üí v88") when the server is ahead**. Tapping a **red** tag runs `forceUpdate` (delete every cache ‚Üí reload ‚Üí SW reinstalls the latest shell from network); a green tag isn't tappable. This makes a phone stuck on the old shell (iOS caches the SW aggressively ‚Äî otherwise needs the app killed/reopened a few times) both diagnosable and fixable in-app. Works the same on Android (Chrome is far less sticky, so it's usually already green there).

- `concert-data.js` ‚Äî **shared concert-program data + the one printed-name ‚Üí roster matcher** (`Concerts.all` / `Concerts.matcher`), hand-transcribed from the PDFs in `programs/` ‚Äî one entry per concert (`id`/`title`/`time`/`venue`/`poi`/`pdf`; **`day` is derived from the id's date**, so a hand-typed heading can't contradict the date the schedule keys on) with each piece's performers as `[name-as-printed, instrument]` pairs (English titles). Each piece has `t` (full program title, movement in `m`, shown on concerts.html) and may carry an optional **`nick`** ‚Äî a match-only nickname alias (the "Ghost" trio, Op. 70 No. 1, whose printed title omits the name) that `dressOf` needs to resolve a slot the sheet labels by nickname; it's not displayed. Its **short rehearsal-style title** (the exact wording the live sheet uses for that piece's rehearsals, e.g. "Bruch Octet" / "Beethoven String Trio", shown on the schedule's brass card *and* the resolved dress card) lives in the **`SHORTS` registry keyed by composer + title** (`shortKey`, "Bruch | String Octet‚Ä¶" ‚Äî not title alone, since generic titles like "Oboe Quartet" / "Clarinet Quartet" would collide across composers) and is resolved onto each piece as `p.s` at load. The registry is deliberately **not inline** so a short name survives a piece being dropped from one concert's final and re-added to another's ‚Äî the entry outlives the move, and re-adding with the same composer + title reattaches `s` (a piece in two concerts, Mendelssohn Op. 87, is named once). Not derivable from `t` (the sheet is inconsistent ‚Äî "Haydn Quartet" one day, "Haydn String Quartet" another; "Bruch Octet" drops "String" but "Beethoven String Trio" keeps it), so keep `SHORTS` in sync with the sheet. **`concert-match-test.js` asserts every non-`brk` piece resolves to an `s`**, so a dropped or mistyped registry key fails loudly instead of shipping a blank/fallback card. Consumers filter `all` by `c.id`'s date prefix and place each card at its own printed `time` (no before/after-18:00 helper anymore ‚Äî a banner's 12-hour time used to misfile the card). **`matcher(people)`** resolves a printed name to a roster person for both concerts.html's kudos chips and app.js's brass test (they were two copies once, and drifted): exact normalized full name; else a full name may match on **surname** (exact or a trailing-letters wobble, "Koffemann"/"Koffeman") when the first names share ‚â•3 leading chars ("Stephen"‚ÜíSteve the DIR, "Preetcharn"‚ÜíPreet) **and** the instruments are compatible (`Roster.instKind`; v/va interchangeable, unknown permissive); else a bare first name matches its unique owner (exact or ‚â•3-char prefix), instrument-checked. Ambiguity, or a surname with no roster counterpart (the 7/8 guest **"Robert Lexer"**, flugelhorn ‚Äî *not* Robert Dembo, clarinet), stays unmatched ‚Äî never a wrong link, never someone else's brass. `scripts/concert-match-test.js` pins all of this. Loaded by `index.html` (before `app.js`) and `concerts.html`; exports on `globalThis` under Node for the harness (app.js still typeof-guards every use). Add a concert's entry here when adding its PDF; when a final replaces a draft, repoint `pdf` + `sw.js` `SHELL`, bump `V`.

Data sources:
- **Schedule** ‚Äî the public, view-only Google Sheet via its **gviz endpoint as JSONP** (`tqx=out:json;responseHandler:cb`). JSONP is deliberate: a plain `fetch` hits CORS, and we only have *view* access so we can't "Publish to web." Don't switch it to `fetch`.
- **Weather** ‚Äî Open-Meteo hourly + daily (`temperature_2m`, `precipitation`, `weathercode`, daily max/min/sun). Primary model is **GeoSphere Austria's `geosphere_seamless`** (AROME 2.5 km for the first ~60 h, ECMWF IFS beyond) ‚Äî it resolves this alpine valley far better than the global best-match blend and still covers the 2-week range; **falls back to Open-Meteo best-match** if geosphere is unavailable (`weatherRange` tries them in order; `fetchWx` skips days with null temps). Each day is tagged with its source and the active one is cited in the weather card ("forecast via ‚Ä¶", `WX_SRC`). CORS-friendly, normal `fetch`.
- **Roster** ‚Äî a *separate* view-only sheet (`ROSTER_SID` in `roster.html`), gviz JSONP, cached offline. Kept out of git on purpose (names + lodging + phone); the page is `noindex` but, like the schedule, still public to anyone with the URL (gviz needs the sheet link-viewable). Now also holds per-person Pieces/Instrument/Hometown/WhatsApp (filled from the repertoire sheet + a phone-number pass).
- **Repertoire / pieces** ‚Äî a *third* view-only sheet (the Group ¬∑ Piece ¬∑ Player 1‚Ä¶8 grid), gviz JSONP, configured in `network.js`; joined with the roster to build the network graph (and the source for the roster's Pieces/Instrument columns).
- **Offline** ‚Äî `app.js` stores each day's parsed schedule + weather in `localStorage` (key `akm-cache`), renders instantly from it on open (stale-while-revalidate), then refreshes if online. Every successful online load re-caches the **whole week**, so a morning open on wifi covers the afternoon offline. Day chips browse cached days.

## Constants (top of app.js)

- `SID` ‚Äî sheet id. Tab naming convention is **`Ddd M/D`**, no leading zeros (e.g. `Mon 6/29`). Confirmed against the live sheet.
- `GID` ‚Äî tab name ‚Üí numeric gid, so the footer "source sheet ‚Üó" link can deep-link to the day's actual tab (`/edit?gid=‚Ä¶#gid=‚Ä¶`). gids can't be fetched cross-origin at runtime, so they're baked in and **auto-maintained** ‚Äî don't hand-edit (see *gid map* below).
- `FEST = ["2026-06-29","2026-07-12"]` ‚Äî drives day chips + the weather range. Day 1 = 6/29.
- `LAT/LON` default to `46.6928/12.8166` and are refined at runtime from the **Kultursaal** POI in `map-data.json` (since the 2026-07-03 relabel that's the Badstub'n concert hall at Klebas 30; the whole village is within forecast tolerance ‚Äî we pin the named concert venue for explainability). `loadPlaces` inverts its `xy` through `meta.bbox`. `TZ = Europe/Vienna`. The laptop is in the Alps, so local = festival time (no timezone math anywhere).
- **Identity** ‚Äî no `MINE`/`ME` constants anymore: the app is **multi-user**. A first-open "Who are you?" picker (one shared URL) stores the canonical roster Name in `localStorage["akm-me"]` (`Roster.me()`/`setMe()`; `null` = never asked ‚Üí auto-open once, `""` = declined ‚Üí never nag; the picked user's brass first name sits right after the **"Schedule" label** in the timeline header (`tl-who`, so whose day you're viewing is clear up top) and both **it** and the footer's "switch user" button reopen the picker ‚Äî the footer no longer repeats the name (redundant with the header), showing a "no one picked ¬∑" nudge only when unpicked). The header's freshness stamp is terse (`@Thu 02:31`, `asofText`) to leave room for the name on a phone; `.tl-head` wraps the meta below on the narrowest screens. **Self-added events are stored per-identity** (`akm-mine:<name>`; identity-less falls back to a bare `akm-mine`), so switching users doesn't bleed events ‚Äî and because an ownerless event otherwise gets silently *adopted* by the next identity that loads (`loadMine`), **adding a new event requires a picked identity**: `openAdd` gates a new add behind the picker (`meSet`/`pendingAdd`, resumes the add after you pick; declining abandons it), while editing/deleting an existing one is ungated. `userCtx()` builds the user from their roster row: group‚Üípiece map from the **Pieces cell's letter prefix** (`"BCE: Dvorak ‚Ä¶ | Faure ‚Ä¶ | Bruch ‚Ä¶"`, letters map positionally), nickname from a "Goes by X" note, hotel, type. `mineOf()` marks a rehearsal cell "yours" when the block's `Group X` label names one of your groups and the cell passes `fits()` + **best token overlap** inside that block: the cell must share a **composer-zone token** (canonical words before the first instrument/ensemble word) and may not cross **ensembles** (quartet‚âÝsextet) or **instrument families** (clarinet‚âÝstring‚âÝpiano; cello‚âàstring so "Schubert Cello Quintet" = D.956; "Casarrubios Piano Trio" = luzAzul matches through the composer zone), and strict argmax keeps same-composer neighbours apart (Brahms Sextet vs Brahms Clarinet Trio in one block). These guards exist because **stale group letters** (week-2 grids before the sheets are updated) otherwise cause *false* brass cards ‚Äî sweep-verified against every posted day. On top of that, **week gating**: the day's eyebrow (Week One/Two) is matched against (a) the user's own week (`W1`/`W2` token in Type ‚Äî a week-1 attendee gets nothing in week 2) and (b) each group's inferred week ‚Äî **a piece shared with any W1-annotated player is a week-1-only group** (matches the pieces sheet's `(W1)` tags); unannotated groups run both weeks. `lessonsOf()` matches lesson-slot student names by first name, "Goes by" nickname, or a **‚â•3-char diminutive prefix** ("Matt" ‚Üí Matthew); when a name fits two lesson-takers ("Steph"), the slot's **coach instrument breaks the tie** (a cellist doesn't take lessons from violin faculty; cello‚âàstring families as elsewhere) and unresolvable ties stay unmatched. F/DIR are skipped as students (they coach ‚Äî AF/TF/MGR take lessons), which also disambiguates the duplicate Tanyas/Stephens. **Type cells can compound** ("F, W1" = week-1-only faculty) ‚Äî all type checks are token-based (`hasTok`), never string equality; roster.html's sort rank and network.js's bold do the same. Guests and anyone without group letters get the generic day. `JASON` remains only to gate the grace-note coda (his easter egg).

Ownership reuses **`myConcertPieces`** (the shared roster matcher) ‚Äî the very test the brass concert card runs ‚Äî so a dress card and its concert card can't disagree about who's performing. The one gap the program can't bridge is a **nickname the sheet uses but the printed title omits** (the "Ghost" trio, Op. 70 No. 1): that piece carries a match-only **`nick`** alias in `concert-data.js`. The card shows the program's short title, so even a mislabelled slot *reads* right. No program cached for the day (or no date) ‚Üí no dress cards ‚Äî the same blank state `mineOf` has when the repertoire's uncached. Matches surface as brass "Dress rehearsal ¬∑ be on time" cards (own playing) carrying a big brass "!" corner mark (`gBang`/`.gbang`) in the Group-letter slot ‚Äî a "don't miss your short window" flag, not a coordination handle; dress slots also count as busy for `freeOf`.

`scripts/build-map.py` (stdlib-only, PEP 723) fetches the Liesing/Klebas bbox from the Overpass API, slims it to roads/buildings/water/land + the festival POIs, projects lat/lon ‚Üí integer metres, and writes `map-data.json`. It's network-tolerant like the gid script: on fetch failure it leaves the existing file untouched and exits 0. (The default `overpass-api.de` endpoint is often overloaded ‚Äî if it 504s/times out, temporarily point `API` at `https://overpass.openstreetmap.fr/api/interpreter` for the run, then revert.)

The **`POIS` list at the top of the script is the single source of truth** for the highlighted places ‚Äî address entries (`hn`+`st`) snap to a real building footprint via `addr:housenumber`/`addr:street`; OSM-named entries (the restaurants, the church) snap to the tagged feature. This is what resolved the prototype's "accepted addresses awaiting OSM anchors." Edit `POIS` (or to refresh geometry), then rerun:

```
python3 scripts/build-map.py    # prints feature + POI counts; WARNs on any unresolved POI
```

**Overpass-blocked?** A full rebuild needs the network, but a *POI-only* edit usually doesn't ‚Äî so `build-map.py` has an **Overpass-free `--offline` mode** (also the automatic fallback when the fetch fails). It reconciles `POIS` against the *already-baked* `map-data.json`: alias changes flow straight through (pure metadata on POIs that already have geometry), and a new **address-anchored** POI is promoted from its baked building ‚Äî every building carries its `a` (e.g. `"Liesing 20"`) + footprint `p`, so it becomes a pin (`xy` = centroid, `fp` = the footprint) and is dropped from `buildings`. So the flow is: edit `POIS`, run `python3 scripts/build-map.py --offline`, then bump `sw.js` `V` since `map-data.json` is precached. Only `osm`/`way`-anchored POIs genuinely need a real Overpass run (the mode WARNs and lists them).

`scripts/build-terrain.py` (uv + PEP 723: Pillow + numpy) bakes the two raster layers, reading the bbox **back from `map-data.json`'s `meta`** so they register pixel-for-pixel with the vector ‚Äî so always run it *after* `build-map.py`:

```
python3 scripts/build-map.py        # vector + labels + meta.bbox
uv run scripts/build-terrain.py     # map-relief.jpg (Terrarium DEM hillshade) + map-aerial.jpg (basemap.at ortho)
```

**Not** in the pre-commit hook (network + slow; town geometry is ~static). `map-data.json` and both JPEGs are committed. `build-map.py` writes `map-data.json` **one feature per line** (`write_json`: compact leaves so a road's int-coord array stays inline, but every road/building/label/POI on its own line) ‚Äî so it greps by name and git-diffs per feature; a plain `json.dump(indent=‚Ä¶)` would explode the coordinate arrays into tens of thousands of lines. **Attribution is required and per-layer** ‚Äî `map.html` swaps it with the active base: Map ‚Üí "¬© OpenStreetMap" (ODbL); Relief ‚Üí "AWS Terrain Tiles"; Aerial ‚Üí "¬© basemap.at ¬∑ CC BY". Keep all three. (Esri/Google aerial imagery can't be bundled offline ‚Äî basemap.at's CC BY orthophoto is why Aerial can be baked in.)

## Gotchas

- **gviz JSONP, not fetch** (CORS + view-only). If it ever returns blank: sheet sharing changed (must stay link-viewable) or that day's tab isn't posted ‚Äî the app shows "not posted yet" rather than erroring.
- **iOS** has no background refresh; it updates on open ‚Äî and since the home-screen card may *resume* (not reload), `app.js` re-pulls on `visibilitychange` and offers **pull-to-refresh** (`#ptr`). Cold start shows a "Loading‚Ä¶" state rather than falsely claiming "not posted." We chose **pure-pull on purpose** ‚Äî push would re-add a scheduled sender, and home-screen web push is also DMA-restricted on EU-region devices.
- **Coda rotation** is deterministic by festival day (sourced tier first), so no state file is needed.
- **Bank integrity** ‚Äî entries are tiered `sourced` / `attributed` / `standard`; never render a quote as more certain than its tier. Quotes are page-cited (Dvo≈ô√°k ‚Üí ≈Ýourek; Bruch ‚Üí Fifield). One landmine: the line *"I should be glad if something occurred to me‚Ä¶"* is **Brahms about Dvo≈ô√°k** (≈Ýourek p. 15), not a Dvo≈ô√°k self-quote ‚Äî it lives in the Brahms entry. Don't "fix" it back. When expanding the bank, prefer omitting a doubtful entry over including a spurious one.
- **`analytics.gs` edits don't deploy themselves** ‚Äî the repo copy of `scripts/analytics.gs` is *reference only*; the live code is the Apps Script bound to Jason's private analytics sheet. Any change here must be hand-pasted into that script **and** re-deployed via **Deploy ‚Üí Manage deployments ‚Üí ‚úé ‚Üí Version: New ‚Üí Deploy** (a plain save does NOT update the live `/exec` URL ‚Äî Apps Script's classic trap). If a redeploy ever mints a *new* URL (e.g. "New deployment" instead of editing the existing one), `URL_` in `ping.js` must be updated + `sw.js` `V` bumped. Two invariants to keep in lockstep across `ping.js` ‚Üî `analytics.gs`: `TOK` (same literal in both), and the uid recipe (SHA-256 over UTF-8, first 4 bytes, lowercase hex ‚Äî `uid()` in ping.js must equal `UID()` in the sheet, or the who‚Üíname key silently stops matching; sanity check: `UID("Jason Sundram") = "70f71792"`). When changing what a ping carries, remember old clients' queued pings flush with the *old* params ‚Äî keep `doGet` tolerant of missing ones. One more trap: a uid whose 8 hex chars are all decimal digits (~2% of names ‚Äî Nathan Meltzer's `51593363` is one) gets **coerced to a number** by `appendRow`, so text-exact lookups against `=UID()` miss it. The `who` column is formatted Plain text to stop future coercion, and the who‚Üíname lookup wraps the value in `IF(ISNUMBER(‚Ä¶), TEXT(‚Ä¶,"00000000"), ‚Ä¶)` to catch already-coerced rows (the `00000000` restores a leading zero Sheets would have eaten). The new **`to`** column (kudos-recipient uid) is the same kind of value, so it needs the **same Plain-text formatting + coercion-safe reverse lookup**.

## Open / nice-to-have

- **Post-festival hardening (before AKM 2027) ‚Äî positional-tuple debt.** The schedule pipeline passes rows around as **positional arrays whose arity kept growing** as features landed this year, and that's the main brittleness to pay down. Concretely: `day.rehearsals` entries are `[start, end, piece, room, coach, tag, grp, partial]` (8 fields); `mineOf`'s *output* is `[s, e, piece, room, coach, tag, composerKey, partial]` ‚Äî note **index 6 means `grp` on input but `composerKey` on output**, and rendering reads them with skip-a-slot destructuring like `[s,e,piece,room,coach,tag,,partial]`. That "meaning-by-position, leave-a-hole" pattern is the highest-odds place a *next* added field silently misaligns. Fix = give these tuples **named fields** (small objects, e.g. `{s,e,piece,room,coach,tag,grp,partial}`) ‚Äî retires ~80% of the risk and makes the next faculty-ish feature boring to add; it's a ~1 hr mechanical refactor, best done on green tests (parser 60/60 + the two live guards) when nothing's live.
