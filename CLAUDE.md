## What this is

A pure-pull **PWA** that shows the day at the AKM Chamber Music Festival: Jason's
rehearsals (room + coach), an hourly weather curve, and a grace-note quote. It fetches the
live schedule and forecast **client-side** — no backend, no scheduled job — and caches the
whole festival week so it works offline once loaded. Hosted on **GitHub Pages**, installed to
the iPhone home screen. Opening it *is* the refresh. Three companion pages (also pure-pull,
offline, SW-precached), reached from a **shared top nav** (`nav.css`) on every page: a
participant **roster**, an offline village **map**, and a co-performance **network** graph.

## Architecture

Static files only. Nothing server-side.

- `index.html` — app shell + all CSS (the design system). Mount point `#app`, a day-chip row, and an online/as-of status line.
- `nav.css` — the **shared top-nav** (`.apphead`) used by every page: the **wordmark** (→ `about.html`, collapses to "AKM" on narrow screens) plus a `nav` of page links (Today / Map / Roster / Network / Concerts) with the current page marked `aria-current`. Uses each page's own palette tokens so it themes (incl. dark) without redefining them; the markup is static per page, this file is the only styling. Replaced the old per-page footer links.
- `app.js` — the brain: fetch → parse → render → cache. Ported 1:1 from the earlier Python runner. Also reads the **shared roster** via `roster-data.js` (`Roster.cached()` / `Roster.pull()` in `refresh`) and derives each person's first name → Notes-URL (`coachMap`), so a **coach's name in a rehearsal/lesson card links to their bio** (`coachLink`; first-name match — faculty URL-holders don't collide; handles a compound `"Gijs/Nathan"` cell; underlined in muted ink, *not* the reserved brass). On a **self-added event** the `with` names each become a **WhatsApp deep-link** when the roster has that person's number (`waName`/`whoWa` resolve the stored `who` token → `Roster.cached()` row → `wa.me/<digits>`, reusing `whoRoster`'s first-name/collision token logic + the `WA_SVG` glyph; the link opens the chat with no pre-filled text, and a name with no number stays plain — never a dead link). The card's delegated click handler already skips any `<a>`, so tapping a name opens WhatsApp instead of the editor. Each rehearsal/coaching/unscheduled card also carries its block's shared **Group letter** (A…E) as a large, faint Fraunces mark in the **bottom-right corner** (`gLetter`), tinted to the card's accent — a coordination handle everyone in the block shares (players *and* coaches), deliberately an oversized corner label rather than a meta-row chip so it can't be misread as a room code (A1/A2). Threaded as a spare tuple index (`mineOf`/`freeOf`/`coachingOf`), so it never collides with `[6]` = composer key. Cards with a letter are stamped `.gcard` (reserves a piece/meta right gutter so nothing runs under the mark; the kicker/＋add row stays full-width). A **(KS) dress-rehearsal** card (`dressOf`, your own playing → brass) carries a big brass **"!"** in the same corner slot instead (`gBang`/`.gbang`) — the block is a short, fixed, easy-to-miss window, so the mark reads as urgency ("be on time"), not a room code or coordination handle. **Informational events** (an open talk, a discussion, a yoga session — e.g. "Emi Presentation in AH", "Multi-Platform Careers Discussion … in AH", "Yoga Session with Elinor in AH") parse into `day.info` (`[start, end, title, room]`) and render for **everyone** regardless of identity as a neutral (never-brass) `.infocard` carrying a muted circled-**"i"** SVG corner mark (`INFO`/`.ginfo`) in the group-letter slot — people have stopped double-checking the sheet, so these get surfaced rather than silently dropped. Detected in `parse` **by shape, not a keyword whitelist**: a prose cell (has lowercase) that names its room *inline* with a word before the "in ROOM" (`Emi Presentation in AH`) — which a standalone "in A4" rehearsal note (nothing before "in"), a coach-tagged rehearsal cell (`- P`/`- C`), an all-caps banner, and any `faculty` item all fail — so a rehearsal, a Faculty Dress, and a meal never masquerade as one. Works in both sheet layouts: time + title in one letter-spaced cell (Sat) or split across cells (Fri). Runs after meals/concerts/dress (they `continue` first) and consumes the row so it can't fall through into a junk rehearsal. A **faculty presenter's name in the title links to their bio** (`linkNames` — any word matching a `coachMap`/`COACHES` key, i.e. a roster URL-holder, becomes the same muted bio link a coach's name gets): "Emi"/"Tanya" resolve (a first-name collision like the two Tanyas lands on the faculty URL-holder for free, since only faculty carry bio URLs), while a non-faculty presenter (Elinore, W1 — no bio URL) stays plain text.
- `roster-data.js` — **shared roster data layer** for the schedule, roster, and map pages. One gviz JSONP pull + one header-tolerant parse (gviz flip-flops between labelling the cols and dumping the header into row 0 — try labels, else row 0) + one `localStorage` cache (`akm-roster`). Also owns the two cross-page identity bits: **`me()`/`setMe()`** (the picked identity, `akm-me`) and **`hotelPoi()`** (roster Hotel value → map POI name; "Obernosterer Apartment" → the "Haus Obernosterer" pin + the Kultursaal apartment — "Obernosterer AirBNB" is its own POI at Liesing 50, so it self-resolves). Also owns **`instKind()`** — the single instrument classifier (abbreviations *and* full names → `v`/`va`/`vc`/`bass`/`piano`/`clarinet`/`oboe`/`v/va`/`""`, unknown never guessed): roster.html's `instLabel`, app.js's `fams` (lesson families), and network.js's Winds grouping all derive from it, so they can't drift (they used to be three prefix-ladders — a "Cello" read as Clarinet on the roster because only that copy lacked the `ce` branch; `scripts/inst-test.js` pins it). network.html loads `roster-data.js` too for this. Exposes `window.Roster` (`SID`/`GID`/`KEY`/`parse`/`cached`/`pull`/`me`/`setMe`/`hotelPoi`/`instKind`); loaded **before** the page scripts. Single source of truth so the gviz-header quirk is fixed once; and since the schedule page (opened far more often) primes the same cache, the roster page renders offline even if it was never opened online.
- `ping.js` — **usage pings**, loaded on every page. Each open (and throttled resume) appends `{ts, page, uid}` to a `localStorage` queue (`akm-pings`, capped) then flushes it via `no-cors` GET to a private Apps Script endpoint (`scripts/analytics.gs`, bound to a **private** analytics sheet in Jason's account — one row per open; the sheet is the dashboard). Offline opens flush on the next online one. `uid` = first 8 hex of SHA-256 of the picked name — no names in the stream; the sheet-side `=UID(name)` formula rebuilds the hash→name key from the roster anytime (stable across roster renumbering, unlike the `#` column). Empty `uid` = no identity picked — the tripwire for strangers hitting the public URL. Reads `localStorage["akm-me"]` directly (no `Roster` dependency — network.html doesn't load roster-data.js). The page name comes from the pathname, so a **directory URL would read as "index"** — pages served at a `/dir/` URL (usage/) set `window.AKM_PAGE` before loading this. Beyond opens, it exposes **`window.AKMPing.event(action, toName, label)`** for **feature-use events**: concerts.html calls `event("kudos", <recipient>, <composer>)` when a kudos chip is tapped, so Jason can tell whether applause is a couple of friends or widespread — and for which piece. An event row carries an **empty `page`** (so both usage crunches — which count opens and filter on a non-empty page — skip it automatically, no crunch/test change), the sender's uid in `who`, the event name in a new **`action`** column, the **recipient's** uid in a new **`to`** column (`toName` hashed with the same recipe, so no names enter the stream and both ends reverse through `=UID()`), and an optional non-PII **`label`** column in plaintext (the composer, URL-encoded on the wire for diacritics); events are unthrottled (each is a real tap) and flush re-reads the queue each pass so an event enqueued mid-flush isn't clobbered. **`URL_` empty = endpoint not deployed: pings queue silently, nothing is sent, nothing errors** — and the backlog flushes once the URL lands. about.html's privacy note deliberately doesn't mention pings yet (Jason wants the wording just right first).
- `sw.js` — service worker; precaches the shell for offline, cache-first for Google Fonts (so the type survives offline), passes cross-origin *data* calls (gviz, open-meteo) through to network. Bump `V` when the shell changes — a new `V` is what evicts the stale cache on activate, so a `SHELL`-file edit that forgets it ships to the repo but never to installed clients (this bit the v77 concert-card rewrite). A warn-only pre-commit lint (`scripts/sw-lint.py`) nags when a commit stages a precached `SHELL` file but leaves `V` unbumped. The **schedule footer prints the installed `V`** next to "switch user" (`verHtml`/`checkVer`/`.swver` — read from `caches.keys()`, since `caches.open(V)` names the cache after `V`, so it can't drift from a hardcoded string): it shows the version *this device actually took*. `checkVer` (on boot + foreground) also fetches the deployed `sw.js` (`?_=`+no-store to dodge both caches) and compares — the tag is **green (`.ok`) when current, red + underlined (`.behind`, "· akm-v87 → v88") when the server is ahead**. Tapping a **red** tag runs `forceUpdate` (delete every cache → reload → SW reinstalls the latest shell from network); a green tag isn't tappable. This makes a phone stuck on the old shell (iOS caches the SW aggressively — otherwise needs the app killed/reopened a few times) both diagnosable and fixable in-app. Works the same on Android (Chrome is far less sticky, so it's usually already green there).
- `manifest.json` + `icon-*.png` — installable/standalone. Icons are rasterized from `icon.svg` (the source of truth) via `scripts/make-icons.sh`; edit the SVG, rerun the script, don't hand-edit the PNGs. `icon.svg` doubles as the in-browser favicon.
- `og.svg` → `og.png` — the 1200×630 **link-preview card** (the icon's dawn scene recomposed wide, wordmark left). Every page's `<head>` carries OG/Twitter meta pointing at the **absolute Pages URL** of `og.png` (WhatsApp/Messenger/Instagram need absolute URLs and won't render SVG). Edit the SVG, rerun `scripts/make-og.sh` — the wordmark is live text, so Fraunces + IBM Plex Mono must be installed locally or it silently falls back to a stock serif. *Not* SW-precached: only link crawlers fetch it.
- `composer-bank.json` — vetted quotes + facts, provenance-tiered.
- `notes.html` — standalone **noindex** archive of **every grace note**, grouped by composer (the quotes/facts that surface one at a time under the schedule, gathered to read in a row). Reads the same `composer-bank.json`, SW-precached, offline. Reached only from the in-app **grace-note coda** — which itself renders for Jason only (`JASON` gate) — so it stays part of his easter egg; the coda's "Grace note" label and each `by`/tag deep-links here (`#<composerKey>`).
- `programs/` — the printed **concert programs** as PDFs (`YYYY-MM-DD-afternoon|evening.pdf`; the 7/11 morning concert is `-morning` but keyed as before-18:00). On the schedule, every `concert-data.js` entry for the shown day renders as a **card** (`concertCard`, one template — the brass/audience differences are computed up front): **brass "Concert · you're performing"** listing the picked user's pieces — by their **short rehearsal-style title** (`p.s`, e.g. "Bruch Octet", so the card reads the same as the day's rehearsal cards; falls back to `composer — full title` if a piece lacks `s`) — when a printed name resolves to *them* through the **shared matcher** (`myConcertPieces` → `Concerts.matcher`, see `concert-data.js` — a program namesake like "Stephen Lustig" vs Steve Buck can never claim someone else's brass), else a cool **"Concert · audience"** card (`.ccard` — brass stays "your own playing"); both carry the **group letter's star counterpart** in the same bottom-right corner (`STAR`/`.gstar` + the shared `.gcard` gutter) — an SVG star (not the ☆/★ glyph, so size + fill are font-independent), tinted brass on the performing card, muted on the audience card. Both carry the venue chip (the sheet banner's live "@ venue" **wins over the baked `poi`** when it names a genuinely different place, so a day-of move reaches users) and a single **"view program"** link into `concerts.html#<id>` (the concert page, which itself links the printed PDF; red `DRAFT` tag while the pdf has the `-draft` suffix). The card is driven by `concert-data.js`, **not** by a sheet CONCERT banner: `timeline()` filters `Concerts.all` by the shown day's date prefix and places each card at the concert's own printed time (`concStart`, "8:00 pm"→"20:00"). Any CONCERT banner in the day's tab for which we hold a program is suppressed (no plain line) so it can't double the card *or* misfile it — the banner's own time is sometimes written 12-hour ("8:00" for 8pm), which used to land the card in the afternoon slot and drop it to a bare "All welcome" line (the Faculty-Concert-7/8 bug). A banner concert with **no** data entry still shows the plain "All welcome" banner line. To add a day's: drop the PDFs in, add the `concert-data.js` entry, add the PDFs to `sw.js` `SHELL` + bump `V` — precached, so the program opens **offline at the venue**; a final replacing a draft = add under the plain name, repoint `pdf` + `SHELL`, delete the draft, bump `V`. (Programs are public handouts — performer names here don't violate the PII rule, which is about lodging + phones.)
- `concert-data.js` — **shared concert-program data + the one printed-name → roster matcher** (`Concerts.all` / `Concerts.matcher`), hand-transcribed from the PDFs in `programs/` — one entry per concert (`id`/`title`/`time`/`venue`/`poi`/`pdf`; **`day` is derived from the id's date**, so a hand-typed heading can't contradict the date the schedule keys on) with each piece's performers as `[name-as-printed, instrument]` pairs (English titles). Each piece has `t` (full program title, movement in `m`, shown on concerts.html) and may carry an optional **`nick`** — a match-only alias the dress resolver (`dressResolve`, shared by student + faculty dress) needs when a slot's wording omits the printed title (the "Ghost" trio, Op. 70 No. 1; the 7/8 Kramers song-set the faculty-dress sheet labels "Dutch Pop Songs"); it's not displayed. Its **short rehearsal-style title** (the exact wording the live sheet uses for that piece's rehearsals, e.g. "Bruch Octet" / "Beethoven String Trio", shown on the schedule's brass card *and* the resolved dress card) lives in the **`SHORTS` registry keyed by composer + title** (`shortKey`, "Bruch | String Octet…" — not title alone, since generic titles like "Oboe Quartet" / "Clarinet Quartet" would collide across composers) and is resolved onto each piece as `p.s` at load. The registry is deliberately **not inline** so a short name survives a piece being dropped from one concert's final and re-added to another's — the entry outlives the move, and re-adding with the same composer + title reattaches `s` (a piece in two concerts, Mendelssohn Op. 87, is named once). Not derivable from `t` (the sheet is inconsistent — "Haydn Quartet" one day, "Haydn String Quartet" another; "Bruch Octet" drops "String" but "Beethoven String Trio" keeps it), so keep `SHORTS` in sync with the sheet. **`concert-match-test.js` asserts every non-`brk` piece resolves to an `s`**, so a dropped or mistyped registry key fails loudly instead of shipping a blank/fallback card. Consumers filter `all` by `c.id`'s date prefix and place each card at its own printed `time` (no before/after-18:00 helper anymore — a banner's 12-hour time used to misfile the card). **`matcher(people)`** resolves a printed name to a roster person for both concerts.html's kudos chips and app.js's brass test (they were two copies once, and drifted): exact normalized full name; else a full name may match on **surname** (exact or a trailing-letters wobble, "Koffemann"/"Koffeman") when the first names share ≥3 leading chars ("Stephen"→Steve the DIR, "Preetcharn"→Preet) **and** the instruments are compatible (`Roster.instKind`; v/va interchangeable, unknown permissive); else a bare first name matches its unique owner (exact or ≥3-char prefix), instrument-checked. Ambiguity, or a surname with no roster counterpart (the 7/8 guest **"Robert Lexer"**, flugelhorn — *not* Robert Dembo, clarinet), stays unmatched — never a wrong link, never someone else's brass. `scripts/concert-match-test.js` pins all of this. Loaded by `index.html` (before `app.js`) and `concerts.html`; exports on `globalThis` under Node for the harness (app.js still typeof-guards every use). Add a concert's entry here when adding its PDF; when a final replaces a draft, repoint `pdf` + `sw.js` `SHELL`, bump `V`.
- `schedule-archive.js` — the **frozen festival snapshot** baked into the repo so the whole festival renders on **any device with no live sheet** (post-festival the schedule sheet may be deleted/moved). `window.Archive = {sched, wx}` mirrors the app's `akm-cache` shape exactly, and `app.js` reads it as a **data floor** (`ARCH`/`schedOf`/`wxOf` — a live or cached day always wins, this fills the gaps), so `mineOf`/`lessonsOf`/dress etc. still personalize over the baked days when a roster is cached and a fresh device sees the generic festival. **Generated, never hand-edited:** `scripts/build-archive.js` pulls each festival day's tab via gviz and parses it through app.js's own `parse()`, and bakes **observed weather** through app.js's exported `normWx` (Open-Meteo ERA5 `archive-api`, falling back to the forecast host's best-match when the archive host is unreachable — the range is all in the past, so that's settled data). One day per line for git-diffs. Loaded in `index.html` before `app.js`; **SW-precached** (`sw.js` `SHELL`) so regenerating it means bumping `V`. Performer/lesson names appear here like the concert programs — the PII rule is lodging + phones, which never enter the schedule.
- `concerts.html` — standalone **noindex** page of **every concert program in brief**, one section per concert (reached from the shared top nav; data from `concert-data.js`). Each piece lists its players; a player whose name resolves to a roster number renders as a **"kudos chip"** — a `.duo` **link** (the whole chip is tappable) styled as a plain name half `.who` (instrument in faint suffix) welded to a warm brass **👏 clap button** `.kudos`, so the chip reads as *applause*, not a contact tap (this replaced the old whole-chip-links-with-a-WhatsApp-glyph design; the green glyph felt too transactional — the clap is just where the eye lands, the whole chip is one big target). It deep-links `wa.me/<number>?text=…` pre-filled with a kudos that **names the piece's composer** ("Kudos on an amazing performance! I loved your `<composer>` 👏👏👏", signed by the picked identity when there is one; the composer is `p.c`, threaded into `nameChip` from the render loop, and drops cleanly to no-mention if a piece lacks one) opening in its own tab. (An earlier tap-to-play applause burst was cut: the click hands straight off to WhatsApp, so it never got screen time.) Numbers resolve **at runtime** from the cached roster (`Roster.cached()`/`pull()`), matched via the **shared `Concerts.matcher`** (see `concert-data.js` — the same rules as the schedule's brass test, so the pages can't drift), so **no phone numbers land in the repo**; unmatched names (the guest choir, the flugelhorn trio) print as a plain `.p` chip, no clap button. Tapping a clap fires a **`ping.js` kudos event** (`AKMPing.event("kudos", <recipient>, <composer>)`, delegated on the stable `#concerts` div; the recipient's canonical name rides a `data-to` attr and the piece's composer a `data-piece` attr) so Jason can see whether the feature is used and by/for whom, and for which piece — sender + recipient by uid only (never a name in the stream), composer in plaintext. The `-draft` pdf suffix drives the red `DRAFT` tag. Venue links to its map POI. SW-precached, offline. (The schedule's own "you're performing" concert card keeps its WhatsApp-glyph links for now — this festive treatment is concerts-page-only.)
- `roster.html` — standalone **noindex** participant roster (reached from the shared top nav; see `nav.css`). Fetch + parse + offline cache come from the shared **`roster-data.js`** (`Roster.pull()` / `Roster.cached()`, key `akm-roster`); this page just sorts + paints. Re-pulls on foreground. The sheet id/tab live in `roster-data.js` (`SID`/`GID`); `GID` points at the cleaned tab where one-person-per-row split the old shared-room rows. The **sheet** carries `# · Name · Instrument · Type · Hotel · Pieces · Notes · Hometown · WhatsApp`; the **displayed table** is the PDF-style hotel-tinted `# · Name · ⓦ · Instr. · Type · Hotel` (hotel→fill palette inline, mirroring the PDF tool) — `#` is tightened to two digits and **ⓦ** is a thin column whose **header is the WhatsApp icon itself** (set in `boot` from `WA_SVG`, no text — plus a sort arrow). **Columns map by header name** (`colMap`), so extra/renamed/reordered columns are safe. **Every visible column header sorts** (toggle asc/desc), **including ⓦ**: `#` numeric (handles half-rows like `17.5`), **Type** by role rank `DIR→MGR→F→AF→TF→W1→W2→G` (**G** = non-playing guests, e.g. family sharing a room), **Instr.** collapsed + ordered to `V, V/VA, VA, VC, Bass, Piano, Clar.` (`instLabel`/`INST_ORDER`: all violin parts→V, VA1→VA, V/VA kept distinct before VA), **ⓦ** by the bare digit string (leading country code groups roughly by region; numberless rows sort last, surfacing who hasn't shared a WhatsApp), others lexical, blanks last. Rows with **From / Pieces / Notes tap to expand** (From = Hometown; URLs in Notes auto-link) — but the **Notes text is Jason's private annotations and renders only when the picked identity is him** (`NOTES_OK`; everyone else sees just From/Pieces, and a notes-only row isn't expandable). URLs *mined* from Notes still work for everyone — a person whose **Notes holds a URL gets their name linked** (`noteUrl`, first URL, trailing punctuation trimmed), same as the schedule's coach bio links. The **WhatsApp glyph** (its own column now, not trailing the name) links `wa.me/<digits>?text=…` pre-filled (`WA_MSG`) for rows that have a number; the picked user's row (`Roster.me()`) gets the brass "you" marker, and the pre-filled greeting signs with their first name. **Roster data is deliberately not in the repo** (PII) — edit the sheet to update.
- `network.html` + `network.js` (+ vendored `d3.v7.min.js`) — standalone **co-performance** graph (who plays with whom). Two views behind a **Rings | Chord** toggle (defaults to Rings): *Chord* = musicians by instrument then name, one colour per instrument family; *Rings* = ego-centric BFS from **the picked user** (`akm-me`; centre → ring 1 direct co-players → rings 2/3; no pick or a non-player → the most-connected musician centres, and the brass mark + "you" legend swatch only appear once someone's picked. Node keeps Jason so the test previews are stable). - `map.html` + `map.js` + `map-data.json` (+ `map-relief.jpg`, `map-aerial.jpg`) — standalone village map of Liesing/Klebas, widened to take in the surrounding peaks (reached from the shared top nav; the footer is now just a caption). Renames flow through the offline reconcile via a **`was`** key in `POIS` (matches the baked pin by old name, keeps geometry, refreshes all metadata), and a point-only pin with an address **adopts its baked building footprint** (how Kultursaal got the Klebas 30 outline) — no baked `mine` anymore, "your base" is runtime-only. Regenerate with `scripts/build-map.py` then `scripts/build-terrain.py` (see *map data* below). This **supersedes** the `AKM-Map.zip` prototype (live `tile.openstreetmap.org` tiles — not offline, not custom, pins only).

Data sources:
- **Schedule** — the public, view-only Google Sheet via its **gviz endpoint as JSONP** (`tqx=out:json;responseHandler:cb`). JSONP is deliberate: a plain `fetch` hits CORS, and we only have *view* access so we can't "Publish to web." Don't switch it to `fetch`.
- **Weather** — Open-Meteo hourly + daily (`temperature_2m`, `precipitation`, `weathercode`, daily max/min/sun). Primary model is **GeoSphere Austria's `geosphere_seamless`** (AROME 2.5 km for the first ~60 h, ECMWF IFS beyond) — it resolves this alpine valley far better than the global best-match blend and still covers the 2-week range; **falls back to Open-Meteo best-match** if geosphere is unavailable (`weatherRange` tries them in order; `fetchWx` skips days with null temps). Each day is tagged with its source and the active one is cited in the weather card ("forecast via …", `WX_SRC`). CORS-friendly, normal `fetch`.
- **Roster** — a *separate* view-only sheet (`ROSTER_SID` in `roster.html`), gviz JSONP, cached offline. Kept out of git on purpose (names + lodging + phone); the page is `noindex` but, like the schedule, still public to anyone with the URL (gviz needs the sheet link-viewable). Now also holds per-person Pieces/Instrument/Hometown/WhatsApp (filled from the repertoire sheet + a phone-number pass).
- **Repertoire / pieces** — a *third* view-only sheet (the Group · Piece · Player 1…8 grid), gviz JSONP, configured in `network.js`; joined with the roster to build the network graph (and the source for the roster's Pieces/Instrument columns).
- **Offline** — `app.js` stores each day's parsed schedule + weather in `localStorage` (key `akm-cache`), renders instantly from it on open (stale-while-revalidate), then refreshes if online. Every successful online load re-caches the **whole week**, so a morning open on wifi covers the afternoon offline. Day chips browse cached days.

## Constants (top of app.js)

- `SID` — sheet id. Tab naming convention is **`Ddd M/D`**, no leading zeros (e.g. `Mon 6/29`). Confirmed against the live sheet.
- `GID` — tab name → numeric gid, so the footer "source sheet ↗" link can deep-link to the day's actual tab (`/edit?gid=…#gid=…`). gids can't be fetched cross-origin at runtime, so they're baked in and **auto-maintained** — don't hand-edit (see *gid map* below).
- `FEST = ["2026-06-29","2026-07-12"]` — drives day chips + the weather range. Day 1 = 6/29.
- `LAT/LON` default to `46.6928/12.8166` and are refined at runtime from the **Kultursaal** POI in `map-data.json` (since the 2026-07-03 relabel that's the Badstub'n concert hall at Klebas 30; the whole village is within forecast tolerance — we pin the named concert venue for explainability). `loadPlaces` inverts its `xy` through `meta.bbox`. `TZ = Europe/Vienna`. The laptop is in the Alps, so local = festival time (no timezone math anywhere).
- **Identity** — no `MINE`/`ME` constants anymore: the app is **multi-user**. A first-open "Who are you?" picker (one shared URL) stores the canonical roster Name in `localStorage["akm-me"]` (`Roster.me()`/`setMe()`; `null` = never asked → auto-open once, `""` = declined → never nag; the picked user's brass first name sits right after the **"Schedule" label** in the timeline header (`tl-who`, so whose day you're viewing is clear up top) and both **it** and the footer's "switch user" button reopen the picker — the footer no longer repeats the name (redundant with the header), showing a "no one picked ·" nudge only when unpicked). The header's freshness stamp is terse (`@Thu 02:31`, `asofText`) to leave room for the name on a phone; `.tl-head` wraps the meta below on the narrowest screens. **Self-added events are stored per-identity** (`akm-mine:<name>`; identity-less falls back to a bare `akm-mine`), so switching users doesn't bleed events — and because an ownerless event otherwise gets silently *adopted* by the next identity that loads (`loadMine`), **adding a new event requires a picked identity**: `openAdd` gates a new add behind the picker (`meSet`/`pendingAdd`, resumes the add after you pick; declining abandons it), while editing/deleting an existing one is ungated. `userCtx()` builds the user from their roster row: group→piece map from the **Pieces cell's letter prefix** (`"BCE: Dvorak … | Faure … | Bruch …"`, letters map positionally), nickname from a "Goes by X" note, hotel, type. `mineOf()` marks a rehearsal cell "yours" when the block's `Group X` label names one of your groups and the cell passes `fits()` + **best token overlap** inside that block: the cell must share a **composer-zone token** (canonical words before the first instrument/ensemble word) and may not cross **ensembles** (quartet≠sextet) or **instrument families** (clarinet≠string≠piano; cello≈string so "Schubert Cello Quintet" = D.956; "Casarrubios Piano Trio" = luzAzul matches through the composer zone), and strict argmax keeps same-composer neighbours apart (Brahms Sextet vs Brahms Clarinet Trio in one block). These guards exist because **stale group letters** (week-2 grids before the sheets are updated) otherwise cause *false* brass cards — sweep-verified against every posted day. On top of that, **week gating**: the day's eyebrow (Week One/Two) is matched against (a) the user's own week (`W1`/`W2` token in Type — a week-1 attendee gets nothing in week 2) and (b) each group's inferred week — **a piece shared with any W1-annotated player is a week-1-only group** (matches the pieces sheet's `(W1)` tags); unannotated groups run both weeks. `lessonsOf()` matches lesson-slot student names by first name, "Goes by" nickname, or a **≥3-char diminutive prefix** ("Matt" → Matthew); when a name fits two lesson-takers ("Steph"), the slot's **coach instrument breaks the tie** (a cellist doesn't take lessons from violin faculty; cello≈string families as elsewhere) and unresolvable ties stay unmatched. F/DIR are skipped as students (they coach — AF/TF/MGR take lessons), which also disambiguates the duplicate Tanyas/Stephens. **Type cells can compound** ("F, W1" = week-1-only faculty) — all type checks are token-based (`hasTok`), never string equality; roster.html's sort rank and network.js's bold do the same. Guests and anyone without group letters get the generic day. `JASON` remains only to gate the grace-note coda (his easter egg).

`app.js` exports `{parse, rowsFrom, mins, norm, despace, evblocks, userCtx, mineOf, lessonsOf, coachingOf, teachingOf, dressOf, facDressOf, facDressSlots, dressResolve, freeOf, …}` under Node and only calls `boot()` in a browser, so it's safe to `require`. Build a gviz-JSON fixture (`{table:{rows:[{c:[{v},…]}]}}`), then derive a user's day: `parse(rowsFrom(fixture))` → `mineOf(day, userCtx(roster, name))` / `lessonsOf(day, ctx)`. Verified behaviors to preserve: chronological sort by clock time (not string — "9:00" must precede "14:30"); meal venue spacing intact ("Mascha Wirt"); `despace` collapses only letter-spaced runs ("L U N C H"→"LUNCH"); **room-header cells can wrap** ("BAND\nROOM") — match/store them whitespace-normalized, or a whole column of rehearsals silently vanishes (this ate every morning BAND ROOM cell — Loeffler, Schubert PT — until 2026-07-03); **the sheet grows rooms mid-festival** (A3 became A4 on 7/3, taking Jason's Dvořák with it) — once a row is a header (≥3 known `ROOMS`), its other short all-caps cells are adopted as rooms too, so `ROOMS` is a seed, not a whitelist (the add-sheet dropdown unions in the day's learned rooms); **a faculty rehearsal can be parked in a room column mid-day** — it goes to `day.fac` (never through the evening machinery, which once fabricated a phantom 10:50 "free rooms" block) and is **never rendered anywhere**: faculty rehearsals aren't open invitations, and Jason won't be the one who sent a crowd to one (the old "Worth sitting in on" list is gone for the same reason — decided 2026-07-03; faculty entries still book rooms for the free-room math); cells can mix **coach lines with notes** ("Dvorak Quartet\nJesus - C (half)\nin A4") — coach lines are collected wherever they sit and "in ROOM" notes dropped; **CONCERT banner rows** ("2 0 : 0 0  C O N C E R T @ venue") parse into `allhands` with a map-linked venue — the final day (Sat 7/11) is *only* concerts, so without this it read as "not posted"; `norm` strips **all** diacritics via NFD (Dvořák/Fauré/Kodály), not a hand-list; **unscheduled blocks are explicit** — `freeOf()` finds every Group slot with no conflicting rehearsal/lesson/self-added event (only on a day the user is actually scheduled) and renders a dashed, deliberately *non-brass* "Unscheduled · Practice Time / Free Reading" card that **taps open the add-sheet pre-filled** with the block's times; **group/piece matching** never cross-matches similar titles in one block (Schumann vs Schubert Piano Trio; luzAzul still resolves from "Casarrubios Piano Trio"); **private-lessons** blocks are matched by content (`"Private Lessons"`), not column — they sit in the dedicated LESSONS column some rows but get shoved into an unused room column (e.g. WERNER) others — and only the slots naming **the picked user** surface, rendered as an emphasized brass card ("Your private lesson · be ready", inferred 30-min slot). The room chip shows the **actual column the block is parked in** (`cols[ci]` → WERNER, where you report), falling back to "LESSONS" when it's in the dedicated column. **Evening practice/reading blocks** are split per start time by `evblocks()` (pre- and post-dinner differ in what's booked) and each renders the rooms left **unbooked** ("Free rooms: A1, A2 …") for grabbing extra rehearsal/reading time — computed as the room universe (`day.rooms`, the most complete header) minus faculty rehearsals, private-lesson rooms, and Closed rooms; the evening cell `kind` (`faculty`/`lessons`/`closed`/`other`) drives this and tolerates pre-upgrade cached days (5th tuple element was a bool); **(KS) Dress Rehearsals** — the final-week pre-concert run-throughs — parse into `day.dress` as unlabelled `[start, end, piece, room]` slots: the banner (`"KS Dress Rehearsals"` / `"Dress Rehearsals in KS"`, + a time range) and the `HH:MM - piece` lists sit on **separate rows** (Thu/Fri) or the **same row** (Sat), and the lists are parked in **arbitrary layout columns that are NOT rooms** — the room is KS, read off the banner text; a list cell is detected by shape (every line `HH:MM - …`), the row is consumed so its cells never fall through into junk rehearsals, empty/unposted slots (Sat's blanks) push nothing, and each slot's end is chained to the next slot's start (last one +15). They're **not** group-tagged (each is one piece's run-through) and the slot wording is the loose live-sheet form, so instead of resolving against the festival-wide repertoire, `dressOf(day, u, date)` treats **that day's concert program as the authority** (a dress rehearsal *is* the run-through for the concert that follows): it matches each slot against the pieces `concert-data.js` holds for the shown date, keyed **by composer**, using ensemble/instrument only to break a tie when a composer plays twice in one day. Two payoffs: (1) a **lone composer that day is trusted through sloppy wording** — Fri 7/10's only Beethoven is the String Trio, so a slot the sheet mislabels "Beethoven Piano Trio" resolves *there* rather than to the E-flat Op. 70 trio that played the night before (the reported bug — it was handing Stephen Lustig and the rest of that trio a phantom Friday card); (2) the **week filter is automatic** — a Week-1 quartet can't sit on a Week-2 program, so the Saturday "Haydn String Quartet" is never ambiguous (only Op. 20 No. 5 is in play; the old whole-repertoire argmax tied the two Haydns and silently *dropped* the card). Ownership reuses **`myConcertPieces`** (the shared roster matcher) — the very test the brass concert card runs — so a dress card and its concert card can't disagree about who's performing. The one gap the program can't bridge is a **nickname the sheet uses but the printed title omits** (the "Ghost" trio, Op. 70 No. 1): that piece carries a match-only **`nick`** alias in `concert-data.js`. The card shows the program's short title, so even a mislabelled slot *reads* right. No program cached for the day (or no date) → no dress cards — the same blank state `mineOf` has when the repertoire's uncached. Matches surface as brass "Dress rehearsal · be on time" cards (own playing) carrying a big brass "!" corner mark (`gBang`/`.gbang`) in the Group-letter slot — a "don't miss your short window" flag, not a coordination handle; dress slots also count as busy for `freeOf`. **Faculty dress rehearsals** (the run-throughs before a **Faculty Concert**, e.g. Fri 7/3 and Wed 7/8) personalize the same way but from a *different* source shape: the sheet writes the whole schedule inline in ONE cell (`"Faculty Dress [in KS] 17:20 Shaw 17:35 Schumann 17:50 Webern 18:05 Faure"`) that `parse` files under `day.evening` (an evening practice block, `kind:"faculty"`) or `day.fac` (a stray faculty column) — where it books its hall for the free-room math but, being a faculty item, is **never shown as an open cell to everyone**. `facDressSlots(day)` pulls the `HH:MM/piece` pairs back out at render (chaining each end to the next slot's start, the last to the block's end; hall read off an `"in ROOM"` note, else the cell's column, else KS), and `facDressOf` runs them through the **shared `dressResolve`** — the same concert-program-authoritative matcher + `myConcertPieces` ownership as the student dress — so only the performing faculty see their slots, and they render as the identical brass "be on time" cards (`day.dress` is the union of both at render). Because it's owned via the concert program, non-performers (faculty *or* student) get nothing — which is exactly why it can be surfaced now that the piece assignments exist: the reason it was deferred was not knowing who played what. `dressResolve`'s composer filter also matches a piece's `nick`, so the loose faculty labels resolve (`"Dutch Pop Songs"` → the Kramers song-set's nick; `"Austrian … Flugelhorn"` → "In die Berg bin i gern" by composer token). Faculty **readings** remain deferred.

## map data (baked, hand-run)

`scripts/build-map.py` (stdlib-only, PEP 723) fetches the Liesing/Klebas bbox from the Overpass API, slims it to roads/buildings/water/land + the festival POIs, projects lat/lon → integer metres, and writes `map-data.json`. It's network-tolerant like the gid script: on fetch failure it leaves the existing file untouched and exits 0. (The default `overpass-api.de` endpoint is often overloaded — if it 504s/times out, temporarily point `API` at `https://overpass.openstreetmap.fr/api/interpreter` for the run, then revert.)

The **`POIS` list at the top of the script is the single source of truth** for the highlighted places — address entries (`hn`+`st`) snap to a real building footprint via `addr:housenumber`/`addr:street`; OSM-named entries (the restaurants, the church) snap to the tagged feature. This is what resolved the prototype's "accepted addresses awaiting OSM anchors." Edit `POIS` (or to refresh geometry), then rerun:

```
python3 scripts/build-map.py    # prints feature + POI counts; WARNs on any unresolved POI
```

**Overpass-blocked?** A full rebuild needs the network, but a *POI-only* edit usually doesn't — so `build-map.py` has an **Overpass-free `--offline` mode** (also the automatic fallback when the fetch fails). It reconciles `POIS` against the *already-baked* `map-data.json`: alias changes flow straight through (pure metadata on POIs that already have geometry), and a new **address-anchored** POI is promoted from its baked building — every building carries its `a` (e.g. `"Liesing 20"`) + footprint `p`, so it becomes a pin (`xy` = centroid, `fp` = the footprint) and is dropped from `buildings`. So the flow is: edit `POIS`, run `python3 scripts/build-map.py --offline`, then bump `sw.js` `V` since `map-data.json` is precached. Only `osm`/`way`-anchored POIs genuinely need a real Overpass run (the mode WARNs and lists them).

`scripts/build-terrain.py` (uv + PEP 723: Pillow + numpy) bakes the two raster layers, reading the bbox **back from `map-data.json`'s `meta`** so they register pixel-for-pixel with the vector — so always run it *after* `build-map.py`:

```
python3 scripts/build-map.py        # vector + labels + meta.bbox
uv run scripts/build-terrain.py     # map-relief.jpg (Terrarium DEM hillshade) + map-aerial.jpg (basemap.at ortho)
```

**Not** in the pre-commit hook (network + slow; town geometry is ~static). `map-data.json` and both JPEGs are committed. `build-map.py` writes `map-data.json` **one feature per line** (`write_json`: compact leaves so a road's int-coord array stays inline, but every road/building/label/POI on its own line) — so it greps by name and git-diffs per feature; a plain `json.dump(indent=…)` would explode the coordinate arrays into tens of thousands of lines. **Attribution is required and per-layer** — `map.html` swaps it with the active base: Map → "© OpenStreetMap" (ODbL); Relief → "AWS Terrain Tiles"; Aerial → "© basemap.at · CC BY". Keep all three. (Esri/Google aerial imagery can't be bundled offline — basemap.at's CC BY orthophoto is why Aerial can be baked in.)

## Gotchas

- **gviz JSONP, not fetch** (CORS + view-only). If it ever returns blank: sheet sharing changed (must stay link-viewable) or that day's tab isn't posted — the app shows "not posted yet" rather than erroring.
- **iOS** has no background refresh; it updates on open — and since the home-screen card may *resume* (not reload), `app.js` re-pulls on `visibilitychange` and offers **pull-to-refresh** (`#ptr`). Cold start shows a "Loading…" state rather than falsely claiming "not posted." We chose **pure-pull on purpose** — push would re-add a scheduled sender, and home-screen web push is also DMA-restricted on EU-region devices.
- **Coda rotation** is deterministic by festival day (sourced tier first), so no state file is needed.
- **Bank integrity** — entries are tiered `sourced` / `attributed` / `standard`; never render a quote as more certain than its tier. Quotes are page-cited (Dvořák → Šourek; Bruch → Fifield). One landmine: the line *"I should be glad if something occurred to me…"* is **Brahms about Dvořák** (Šourek p. 15), not a Dvořák self-quote — it lives in the Brahms entry. Don't "fix" it back. When expanding the bank, prefer omitting a doubtful entry over including a spurious one.
- **`analytics.gs` edits don't deploy themselves** — the repo copy of `scripts/analytics.gs` is *reference only*; the live code is the Apps Script bound to Jason's private analytics sheet. Any change here must be hand-pasted into that script **and** re-deployed via **Deploy → Manage deployments → ✎ → Version: New → Deploy** (a plain save does NOT update the live `/exec` URL — Apps Script's classic trap). If a redeploy ever mints a *new* URL (e.g. "New deployment" instead of editing the existing one), `URL_` in `ping.js` must be updated + `sw.js` `V` bumped. Two invariants to keep in lockstep across `ping.js` ↔ `analytics.gs`: `TOK` (same literal in both), and the uid recipe (SHA-256 over UTF-8, first 4 bytes, lowercase hex — `uid()` in ping.js must equal `UID()` in the sheet, or the who→name key silently stops matching; sanity check: `UID("Jason Sundram") = "70f71792"`). When changing what a ping carries, remember old clients' queued pings flush with the *old* params — keep `doGet` tolerant of missing ones. One more trap: a uid whose 8 hex chars are all decimal digits (~2% of names — Nathan Meltzer's `51593363` is one) gets **coerced to a number** by `appendRow`, so text-exact lookups against `=UID()` miss it. The `who` column is formatted Plain text to stop future coercion, and the who→name lookup wraps the value in `IF(ISNUMBER(…), TEXT(…,"00000000"), …)` to catch already-coerced rows (the `00000000` restores a leading zero Sheets would have eaten). The new **`to`** column (kudos-recipient uid) is the same kind of value, so it needs the **same Plain-text formatting + coercion-safe reverse lookup**.

## Open / nice-to-have

- **Post-festival hardening (before AKM 2027) — positional-tuple debt.** The schedule pipeline passes rows around as **positional arrays whose arity kept growing** as features landed this year, and that's the main brittleness to pay down. Concretely: `day.rehearsals` entries are `[start, end, piece, room, coach, tag, grp, partial]` (8 fields); `mineOf`'s *output* is `[s, e, piece, room, coach, tag, composerKey, partial]` — note **index 6 means `grp` on input but `composerKey` on output**, and rendering reads them with skip-a-slot destructuring like `[s,e,piece,room,coach,tag,,partial]`. That "meaning-by-position, leave-a-hole" pattern is the highest-odds place a *next* added field silently misaligns. Fix = give these tuples **named fields** (small objects, e.g. `{s,e,piece,room,coach,tag,grp,partial}`) — retires ~80% of the risk and makes the next faculty-ish feature boring to add; it's a ~1 hr mechanical refactor, best done on green tests (parser 83/83 + the two live guards) when nothing's live.
