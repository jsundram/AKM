# CLAUDE.md — AKM briefing

Context for working on this project in Claude Code. Read this first.

## What this is

A pure-pull **PWA** that shows the day at the AKM Chamber Music Festival: Jason's
rehearsals (room + coach), an hourly weather curve, and a grace-note quote. It fetches the
live schedule and forecast **client-side** — no backend, no scheduled job — and caches the
whole festival week so it works offline once loaded. Hosted on **GitHub Pages**, installed to
the iPhone home screen. Opening it *is* the refresh. Three footer-linked companion pages
(also pure-pull, offline, SW-precached): a participant **roster**, an offline village **map**,
and a co-performance **network** graph.

## Architecture

Static files only. Nothing server-side.

- `index.html` — app shell + all CSS (the design system). Mount point `#app`, a day-chip row, and an online/as-of status line.
- `app.js` — the brain: fetch → parse → render → cache. Ported 1:1 from the earlier Python runner. Also reads the **shared roster** via `roster-data.js` (`Roster.cached()` / `Roster.pull()` in `refresh`) and derives each person's first name → Notes-URL (`coachMap`), so a **coach's name in a rehearsal/lesson card links to their bio** (`coachLink`; first-name match — faculty URL-holders don't collide; handles a compound `"Gijs/Nathan"` cell; underlined in muted ink, *not* the reserved brass).
- `roster-data.js` — **shared roster data layer** for the schedule, roster, and map pages. One gviz JSONP pull + one header-tolerant parse (gviz flip-flops between labelling the cols and dumping the header into row 0 — try labels, else row 0) + one `localStorage` cache (`akm-roster`). Also owns the two cross-page identity bits: **`me()`/`setMe()`** (the picked identity, `akm-me`) and **`hotelPoi()`** (roster Hotel value → map POI name; "Obernosterer Apartment" → the "Haus Obernosterer" pin + the Kultursaal apartment — "Obernosterer AirBNB" is its own POI at Liesing 50, so it self-resolves). Also owns **`instKind()`** — the single instrument classifier (abbreviations *and* full names → `v`/`va`/`vc`/`bass`/`piano`/`clarinet`/`oboe`/`v/va`/`""`, unknown never guessed): roster.html's `instLabel`, app.js's `fams` (lesson families), and network.js's Winds grouping all derive from it, so they can't drift (they used to be three prefix-ladders — a "Cello" read as Clarinet on the roster because only that copy lacked the `ce` branch; `scripts/inst-test.js` pins it). network.html loads `roster-data.js` too for this. Exposes `window.Roster` (`SID`/`GID`/`KEY`/`parse`/`cached`/`pull`/`me`/`setMe`/`hotelPoi`/`instKind`); loaded **before** the page scripts. Single source of truth so the gviz-header quirk is fixed once; and since the schedule page (opened far more often) primes the same cache, the roster page renders offline even if it was never opened online.
- `ping.js` — **usage pings**, loaded on every page. Each open (and throttled resume) appends `{ts, page, uid}` to a `localStorage` queue (`akm-pings`, capped) then flushes it via `no-cors` GET to a private Apps Script endpoint (`scripts/analytics.gs`, bound to a **private** analytics sheet in Jason's account — one row per open; the sheet is the dashboard). Offline opens flush on the next online one. `uid` = first 8 hex of SHA-256 of the picked name — no names in the stream; the sheet-side `=UID(name)` formula rebuilds the hash→name key from the roster anytime (stable across roster renumbering, unlike the `#` column). Empty `uid` = no identity picked — the tripwire for strangers hitting the public URL. Reads `localStorage["akm-me"]` directly (no `Roster` dependency — network.html doesn't load roster-data.js). **`URL_` empty = endpoint not deployed: pings queue silently, nothing is sent, nothing errors** — and the backlog flushes once the URL lands. about.html's privacy note deliberately doesn't mention pings yet (Jason wants the wording just right first).
- `sw.js` — service worker; precaches the shell for offline, cache-first for Google Fonts (so the type survives offline), passes cross-origin *data* calls (gviz, open-meteo) through to network. Bump `V` when the shell changes.
- `manifest.json` + `icon-*.png` — installable/standalone. Icons are rasterized from `icon.svg` (the source of truth) via `scripts/make-icons.sh`; edit the SVG, rerun the script, don't hand-edit the PNGs. `icon.svg` doubles as the in-browser favicon.
- `og.svg` → `og.png` — the 1200×630 **link-preview card** (the icon's dawn scene recomposed wide, wordmark left). Every page's `<head>` carries OG/Twitter meta pointing at the **absolute Pages URL** of `og.png` (WhatsApp/Messenger/Instagram need absolute URLs and won't render SVG). Edit the SVG, rerun `scripts/make-og.sh` — the wordmark is live text, so Fraunces + IBM Plex Mono must be installed locally or it silently falls back to a stock serif. *Not* SW-precached: only link crawlers fetch it.
- `composer-bank.json` — vetted quotes + facts, provenance-tiered.
- `programs/` — the printed **concert programs** as PDFs (`YYYY-MM-DD-afternoon|evening.pdf`). `PROGRAMS` in `app.js` (date → `{aft,eve}`, split at 18:00 so a sheet time shift doesn't misfile the link) appends a muted "program ↗" link to the day's concert rows. To add a day's: drop the PDFs in, add the `PROGRAMS` entry, add them to `sw.js` `SHELL` + bump `V` — precached, so the program opens **offline at the venue**. (Programs are public handouts — performer names here don't violate the PII rule, which is about lodging + phones.)
- `roster.html` — standalone **noindex** participant roster (footer-linked; "← Schedule" back). Fetch + parse + offline cache come from the shared **`roster-data.js`** (`Roster.pull()` / `Roster.cached()`, key `akm-roster`); this page just sorts + paints. Re-pulls on foreground. The sheet id/tab live in `roster-data.js` (`SID`/`GID`); `GID` points at the cleaned tab where one-person-per-row split the old shared-room rows. The **sheet** carries `# · Name · Instrument · Type · Hotel · Pieces · Notes · Hometown · WhatsApp`; the **displayed table** is the PDF-style hotel-tinted `# · Name · ⓦ · Instr. · Type · Hotel` (hotel→fill palette inline, mirroring the PDF tool) — `#` is tightened to two digits and **ⓦ** is a thin column whose **header is the WhatsApp icon itself** (set in `boot` from `WA_SVG`, no text). **Columns map by header name** (`colMap`), so extra/renamed/reordered columns are safe. **Every visible column header sorts** (toggle asc/desc): `#` numeric (handles half-rows like `17.5`), **Type** by role rank `DIR→MGR→F→AF→TF→W1→W2→G` (**G** = non-playing guests, e.g. family sharing a room), **Instr.** collapsed + ordered to `V, V/VA, VA, VC, Bass, Piano, Clar.` (`instLabel`/`INST_ORDER`: all violin parts→V, VA1→VA, V/VA kept distinct before VA), others lexical, blanks last. Rows with **From / Pieces / Notes tap to expand** (From = Hometown; URLs in Notes auto-link) — but the **Notes text is Jason's private annotations and renders only when the picked identity is him** (`NOTES_OK`; everyone else sees just From/Pieces, and a notes-only row isn't expandable). URLs *mined* from Notes still work for everyone — a person whose **Notes holds a URL gets their name linked** (`noteUrl`, first URL, trailing punctuation trimmed), same as the schedule's coach bio links. The **WhatsApp glyph** (its own column now, not trailing the name) links `wa.me/<digits>?text=…` pre-filled (`WA_MSG`) for rows that have a number; the picked user's row (`Roster.me()`) gets the brass "you" marker, and the pre-filled greeting signs with their first name. Footer also links **Network** and **Edit sheet ↗** (deep-links to the tab via `Roster.GID`). **Roster data is deliberately not in the repo** (PII) — edit the sheet to update.
- `network.html` + `network.js` (+ vendored `d3.v7.min.js`) — standalone **co-performance** graph (who plays with whom). Joins the **pieces/repertoire sheet** (Group · Piece · Player 1…8) with the roster (Name · Instrument · Type) via gviz JSONP, builds a co-occurrence graph (edge = a shared piece), caches in `localStorage`, re-pulls on foreground. Two views behind a **Rings | Chord** toggle (defaults to Rings): *Chord* = musicians by instrument then name, one colour per instrument family; *Rings* = ego-centric BFS from **the picked user** (`akm-me`; centre → ring 1 direct co-players → rings 2/3; no pick or a non-player → the most-connected musician centres, and the brass mark + "you" legend swatch only appear once someone's picked. Node keeps Jason so the test previews are stable). Pieces-sheet names (first-name-only, `(W1)` tags) reconcile to canonical roster names by **exact or unambiguous first-name match only — no alias table**; names are fixed at the source (the sheet, keeping participant names out of this public repo), and `network-test.js`'s `0 unmatched` assertion guards it. `scripts/network-test.js` is the Node harness; its SVG previews are **gitignored** (they bake in PII).
- `about.html` — standalone **About** page and the **invite link Jason shares** (decided 2026-07-04: the root stays the app — `manifest.json`'s `start_url:"."` means installed home-screen copies open the root many times a day, so the read-daily page keeps the root and the read-once page is the link you send; don't swap them). Reached in-app via the **wordmark** in every page's header (which used to link to alpenkammermusik.com; that link now lives here). Plain first-person copy in Jason's voice — **no self-quote, no marketing gloss** (he wrote it; quoting himself is out) — with a prominent **"Open today's schedule →" CTA** right after the intro, terse section labels (Your day / Offline / Install / More AKM / Privacy / Bugs), links to the festival site / shared Google Photos album / Instagram, and a colophon with data credits. Has OG/share-card meta (`og.png`, hand-made). The "contact Jason" link deep-links to WhatsApp with the number **derived at runtime from the cached roster** (`Roster.cached()`, falls back to the roster page) — no phone number in the repo, per the PII rule. Deliberately does *not* mention the grace notes (easter egg) or the Notes-column visibility rule.
- `map.html` + `map.js` + `map-data.json` (+ `map-relief.jpg`, `map-aerial.jpg`) — standalone village map of Liesing/Klebas, widened to take in the surrounding peaks (linked from the footer; "← Today" back). A **baked OSM vector basemap** drawn as a custom SVG in the alpine palette — no tiles, no map libs, **fully offline** (every file is SW-precached). `map.js` loads `map-data.json` (geometry in integer metres), paints layers (land/water/roads/buildings + water/hamlet labels — the **Gail**, the **Badeteich** swimming pond, etc.), highlights the festival POIs (lodging/venue/food footprints + pins), and does its own pan/zoom (transform on `#scene`, clipped to the bbox, markers repositioned in screen space). A **Map · Relief · Aerial** segmented control swaps the base: *Relief* = baked hillshade (`map-relief.jpg`), *Aerial* = baked 30 cm orthophoto (`map-aerial.jpg`); the vector roads/labels/pins overlay all three, and the attribution line swaps per layer. Lodging names mirror `roster.html`; **the one warm-brass POI is "your base"** — `myBase()` clears the baked `mine` flag and re-points it at the picked user's roster Hotel via `Roster.hotelPoi()` (no pick / unmapped lodging → no brass pin), so `map.html` loads `roster-data.js` too. A live **blue "you are here" dot** (geolocation `watchPosition`, projected with the same equirectangular formula as `build-map.py`, read from `meta.bbox`) sits in a faint accuracy ring, repositioned in screen space like the pins; a **◎ locate** control requests permission + recenters. POI pins can carry **aliases** — the schedule's room codes surfaced under the label (A1–A4/AH → **Akademie**, the Volksmusik Akademie at Liesing 15; KS + "Badstubn" → **Kultursaal**, the Badstub'n hall at Klebas 30 — every schedule "KS"/"Kultursaal" means *that* building; WERNER → Liesing 30). A pin with a second role carries `also` and renders **split-colour** — pin *and* footprint (SVG gradient built per `cat-also` pair, stops themed via CSS vars): Lesachtalerhof = lodging+food, Kultursaal = venue+lodging (the Badstub'n is the concert venue *and* the Kultursaal apartment); the legend carries a swatch per combo. Renames flow through the offline reconcile via a **`was`** key in `POIS` (matches the baked pin by old name, keeps geometry, refreshes all metadata), and a point-only pin with an address **adopts its baked building footprint** (how Kultursaal got the Klebas 30 outline) — no baked `mine` anymore, "your base" is runtime-only. Regenerate with `scripts/build-map.py` then `scripts/build-terrain.py` (see *map data* below). This **supersedes** the `AKM-Map.zip` prototype (live `tile.openstreetmap.org` tiles — not offline, not custom, pins only).

Data sources:
- **Schedule** — the public, view-only Google Sheet via its **gviz endpoint as JSONP** (`tqx=out:json;responseHandler:cb`). JSONP is deliberate: a plain `fetch` hits CORS, and we only have *view* access so we can't "Publish to web." Don't switch it to `fetch`.
- **Weather** — Open-Meteo hourly + daily (`temperature_2m`, `precipitation`, `weathercode`, daily max/min/sun). Primary model is **GeoSphere Austria's `geosphere_seamless`** (AROME 2.5 km for the first ~60 h, ECMWF IFS beyond) — it resolves this alpine valley far better than the global best-match blend and still covers the 2-week range; **falls back to Open-Meteo best-match** if geosphere is unavailable (`weatherRange` tries them in order; `fetchWx` skips days with null temps). Each day is tagged with its source and the active one is cited in the weather card ("forecast via …", `WX_SRC`). CORS-friendly, normal `fetch`.
- **Roster** — a *separate* view-only sheet (`ROSTER_SID` in `roster.html`), gviz JSONP, cached offline. Kept out of git on purpose (names + lodging + phone); the page is `noindex` but, like the schedule, still public to anyone with the URL (gviz needs the sheet link-viewable). Now also holds per-person Pieces/Instrument/Hometown/WhatsApp (filled from the repertoire sheet + a phone-number pass).
- **Repertoire / pieces** — a *third* view-only sheet (the Group · Piece · Player 1…8 grid), gviz JSONP, configured in `network.js`; joined with the roster to build the network graph (and the source for the roster's Pieces/Instrument columns).
- **Offline** — `app.js` stores each day's parsed schedule + weather in `localStorage` (key `akm-cache`), renders instantly from it on open (stale-while-revalidate), then refreshes if online. Every successful online load re-caches the **whole week**, so a morning open on wifi covers the afternoon offline. Day chips browse cached days.

## Constants (top of app.js)

- `SID` — sheet id. Tab naming convention is **`Ddd M/D`**, no leading zeros (e.g. `Mon 6/29`). Confirmed against the live sheet.
- `GID` — tab name → numeric gid, so the footer "source sheet ↗" link can deep-link to the day's actual tab (`/edit?gid=…#gid=…`). gids can't be fetched cross-origin at runtime, so they're baked in and **auto-maintained** — don't hand-edit (see *gid map* below).
- `FEST = ["2026-06-29","2026-07-12"]` — drives day chips + the weather range. Day 1 = 6/29.
- `LAT/LON` default to `46.6928/12.8166` and are refined at runtime from the **Kultursaal** POI in `map-data.json` (since the 2026-07-03 relabel that's the Badstub'n concert hall at Klebas 30; the whole village is within forecast tolerance — we pin the named concert venue for explainability). `loadPlaces` inverts its `xy` through `meta.bbox`. `TZ = Europe/Vienna`. The laptop is in the Alps, so local = festival time (no timezone math anywhere).
- **Identity** — no `MINE`/`ME` constants anymore: the app is **multi-user**. A first-open "Who are you?" picker (one shared URL) stores the canonical roster Name in `localStorage["akm-me"]` (`Roster.me()`/`setMe()`; `null` = never asked → auto-open once, `""` = declined → never nag; "for *first* · switch" footer reopens it). `userCtx()` builds the user from their roster row: group→piece map from the **Pieces cell's letter prefix** (`"BCE: Dvorak … | Faure … | Bruch …"`, letters map positionally), nickname from a "Goes by X" note, hotel, type. `mineOf()` marks a rehearsal cell "yours" when the block's `Group X` label names one of your groups and the cell passes `fits()` + **best token overlap** inside that block: the cell must share a **composer-zone token** (canonical words before the first instrument/ensemble word) and may not cross **ensembles** (quartet≠sextet) or **instrument families** (clarinet≠string≠piano; cello≈string so "Schubert Cello Quintet" = D.956; "Casarrubios Piano Trio" = luzAzul matches through the composer zone), and strict argmax keeps same-composer neighbours apart (Brahms Sextet vs Brahms Clarinet Trio in one block). These guards exist because **stale group letters** (week-2 grids before the sheets are updated) otherwise cause *false* brass cards — sweep-verified against every posted day. On top of that, **week gating**: the day's eyebrow (Week One/Two) is matched against (a) the user's own week (`W1`/`W2` token in Type — a week-1 attendee gets nothing in week 2) and (b) each group's inferred week — **a piece shared with any W1-annotated player is a week-1-only group** (matches the pieces sheet's `(W1)` tags); unannotated groups run both weeks. `lessonsOf()` matches lesson-slot student names by first name, "Goes by" nickname, or a **≥3-char diminutive prefix** ("Matt" → Matthew); when a name fits two lesson-takers ("Steph"), the slot's **coach instrument breaks the tie** (a cellist doesn't take lessons from violin faculty; cello≈string families as elsewhere) and unresolvable ties stay unmatched. F/DIR are skipped as students (they coach — AF/TF/MGR take lessons), which also disambiguates the duplicate Tanyas/Stephens. **Type cells can compound** ("F, W1" = week-1-only faculty) — all type checks are token-based (`hasTok`), never string equality; roster.html's sort rank and network.js's bold do the same. Guests and anyone without group letters get the generic day. `JASON` remains only to gate the grace-note coda (his easter egg).

## Conventions

Jason writes terse, DRY, idiomatic JS — short names, minimal comments (comments explain *why*, not *what*), Bostock-flavored but modern and readable. Match that. Same for any Python in `archive/` (uv + PEP 723 inline metadata, short names).

## Design system (don't drift)

Alpine palette; Fraunces (display) / IBM Plex Mono (data) / Inter (body) — **loaded from Google Fonts** in `index.html` (`<link>` + preconnect) and SW-cached; they are *not* iOS system fonts, so without the link the phone silently falls back to Times/Courier. `viewport-fit=cover` + `env(safe-area-inset-*)` padding keeps content clear of the Dynamic Island / home indicator in standalone mode. **Warm brass cards mean "the picked user's own playing" — their scheduled rehearsals/lessons *and* their self-added personal events** (the latter set apart by a dashed edge, `.self`, not a different colour). Never repurpose the warm accent for anything that isn't theirs to play. The temperature curve (cool→warm gradient, H/L dots, shaded SHOWERS band, dotted sunrise/sunset verticals with a ☀︎ glyph + times flanking the 12p tick) is the signature in-app element. The forecast summary distinguishes **Thunderstorms / Showers / Drizzle** (measurable precip below the 0.2mm/h showers threshold) **/ sky** (Clear · Mostly clear · Partly cloudy · Overcast · Fog, from the daily weather code) — no "wettest window" footer. The app icon/favicon is a flat layered Lesachtal dawn scene (`icon.svg`): warm sun — the one nod to the brass accent — snow-capped hero peak, hazy side ranges, rolling meadows. Full light + dark via `prefers-color-scheme`, contrasts verified WCAG AA+. Edit CSS in `index.html`; render functions in `app.js` emit matching classes.

## Test

**`TESTING.md`** holds the full pre-share checklist: the automated battery, the whole-festival sweep recipe, and the manual on-device suite (upgrade path, fresh-device NUX, identity spot-checks). Summary of the automated side:

The parser has a Node harness (pure functions are exported when `app.js` is required outside a browser):

```
node archive/parser-test.js   # passes 36/36 on the real Tuesday grid (incl. morning all-hands above the room grid, a private lesson, per-block evening free rooms, multi-user piece matching, stale-letter false-positive guards, coach-instrument lesson tie-breaks, week gating, the wrapped room header, concert banner rows, header-learned new rooms, and daytime faculty cells)
```

That harness is **offline + synthetic**, so it runs anywhere — but by construction it can't see the *live sheets drifting from what the code expects* (a roster `Pieces` reformat once dropped the group-prefix `userCtx` parsed, silently blanking **every** user's schedule while these fixtures stayed green). Two **live** smoke tests close that gap by pulling the real sheets (public gviz, no auth):

```
node scripts/network-test.js    # the pieces↔roster co-performance join (0 unmatched); SVG previews gitignored (PII)
node scripts/schedule-test.js   # the schedule personalizes — mineOf non-empty + nobody over-claims — on a wk-1 and a wk-2 grid
```

Both are **network-gated: they skip (`exit 0`), never fail, when the sheets are unreachable** (offline, or a locked-down cloud sandbox). There are deliberately *no committed fixtures* to fall back to — the roster is PII. So they assert when data is reachable and stay quiet otherwise.

`app.js` exports `{parse, rowsFrom, mins, norm, despace, evblocks, userCtx, mineOf, lessonsOf}` under Node and only calls `boot()` in a browser, so it's safe to `require`. Build a gviz-JSON fixture (`{table:{rows:[{c:[{v},…]}]}}`), then derive a user's day: `parse(rowsFrom(fixture))` → `mineOf(day, userCtx(roster, name))` / `lessonsOf(day, ctx)`. Verified behaviors to preserve: chronological sort by clock time (not string — "9:00" must precede "14:30"); meal venue spacing intact ("Mascha Wirt"); `despace` collapses only letter-spaced runs ("L U N C H"→"LUNCH"); **room-header cells can wrap** ("BAND\nROOM") — match/store them whitespace-normalized, or a whole column of rehearsals silently vanishes (this ate every morning BAND ROOM cell — Loeffler, Schubert PT — until 2026-07-03); **the sheet grows rooms mid-festival** (A3 became A4 on 7/3, taking Jason's Dvořák with it) — once a row is a header (≥3 known `ROOMS`), its other short all-caps cells are adopted as rooms too, so `ROOMS` is a seed, not a whitelist (the add-sheet dropdown unions in the day's learned rooms); **a faculty rehearsal can be parked in a room column mid-day** — it goes to `day.fac` (never through the evening machinery, which once fabricated a phantom 10:50 "free rooms" block) and is **never rendered anywhere**: faculty rehearsals aren't open invitations, and Jason won't be the one who sent a crowd to one (the old "Worth sitting in on" list is gone for the same reason — decided 2026-07-03; faculty entries still book rooms for the free-room math); cells can mix **coach lines with notes** ("Dvorak Quartet\nJesus - C (half)\nin A4") — coach lines are collected wherever they sit and "in ROOM" notes dropped; **CONCERT banner rows** ("2 0 : 0 0  C O N C E R T @ venue") parse into `allhands` with a map-linked venue — the final day (Sat 7/11) is *only* concerts, so without this it read as "not posted"; `norm` strips **all** diacritics via NFD (Dvořák/Fauré/Kodály), not a hand-list; **unscheduled blocks are explicit** — `freeOf()` finds every Group slot with no conflicting rehearsal/lesson/self-added event (only on a day the user is actually scheduled) and renders a dashed, deliberately *non-brass* "Unscheduled · Practice Time / Free Reading" card that **taps open the add-sheet pre-filled** with the block's times; **group/piece matching** never cross-matches similar titles in one block (Schumann vs Schubert Piano Trio; luzAzul still resolves from "Casarrubios Piano Trio"); **private-lessons** blocks are matched by content (`"Private Lessons"`), not column — they sit in the dedicated LESSONS column some rows but get shoved into an unused room column (e.g. WERNER) others — and only the slots naming **the picked user** surface, rendered as an emphasized brass card ("Your private lesson · be ready", inferred 30-min slot). The room chip shows the **actual column the block is parked in** (`cols[ci]` → WERNER, where you report), falling back to "LESSONS" when it's in the dedicated column. **Evening practice/reading blocks** are split per start time by `evblocks()` (pre- and post-dinner differ in what's booked) and each renders the rooms left **unbooked** ("Free rooms: A1, A2 …") for grabbing extra rehearsal/reading time — computed as the room universe (`day.rooms`, the most complete header) minus faculty rehearsals, private-lesson rooms, and Closed rooms; the evening cell `kind` (`faculty`/`lessons`/`closed`/`other`) drives this and tolerates pre-upgrade cached days (5th tuple element was a bool).

GitHub Pages, `main` / root. Relative paths throughout, so it works as a project page (`<you>.github.io/AKM/`). After deploy, open the URL **online once** so it caches, then Add to Home Screen. ## gid map (auto-maintained)

`scripts/update-gids.py` probes the sheet's `htmlview` (which embeds every `name → gid`) and rewrites the `GID` literal in `app.js`. A **pre-commit hook** (`.githooks/pre-commit`) runs it on every commit and re-stages `app.js`, so the map self-heals when tabs are added/rebuilt. It's network-tolerant: if the fetch fails (offline) it keeps the existing map rather than blocking the commit — so you may commit a stale map offline; the next online commit fixes it.

Enable the hook once per clone:

```
git config core.hooksPath .githooks
```

Run it by hand anytime: `python3 scripts/update-gids.py`. gids are stable across tab rename/reorder; only a from-scratch rebuild of the sheet changes them.

## buildlog (meta dashboard, laptop-only refresh)

`buildlog/` is the build-time dashboard (Jason's hours three ways vs Claude's, laptop vs cloud, commits, LOC growth) at `/buildlog/` — noindex, **not** SW-precached, no roster data. Every number is computed in-page from one embedded `DATA` blob, so refreshing = re-baking that blob: `buildlog/crunch.py` derives it from the **local Claude Code transcripts** (`~/.claude/projects/…AKM/*.jsonl`) + git history. Transcripts never leave this laptop, so **a cloud session cannot refresh this page** — and time spent on *other* machines isn't measured (their commits/LOC still count; cloud sessions are reconstructed from `Claude-Session` commit trailers). `buildlog/update.py` runs crunch → splice → commit → push; it's idempotent (skips when only the generated-timestamp moved), network-tolerant, and self-expires after 2026-07-13. A launchd job runs it nightly at 23:45 (missed runs fire on next wake):

```
launchctl bootstrap gui/501 ~/Library/LaunchAgents/com.akm.buildlog.plist   # install
launchctl bootout gui/501/com.akm.buildlog                                  # remove
```

Log: `~/Library/Logs/akm-buildlog.log`. Run by hand anytime: `python3 buildlog/update.py`.

## map data (baked, hand-run)

`scripts/build-map.py` (stdlib-only, PEP 723) fetches the Liesing/Klebas bbox from the Overpass API, slims it to roads/buildings/water/land + the festival POIs, projects lat/lon → integer metres, and writes `map-data.json`. It's network-tolerant like the gid script: on fetch failure it leaves the existing file untouched and exits 0. (The default `overpass-api.de` endpoint is often overloaded — if it 504s/times out, temporarily point `API` at `https://overpass.openstreetmap.fr/api/interpreter` for the run, then revert.)

The **`POIS` list at the top of the script is the single source of truth** for the highlighted places — address entries (`hn`+`st`) snap to a real building footprint via `addr:housenumber`/`addr:street`; OSM-named entries (the restaurants, the church) snap to the tagged feature. This is what resolved the prototype's "accepted addresses awaiting OSM anchors." Edit `POIS` (or to refresh geometry), then rerun:

```
python3 scripts/build-map.py    # prints feature + POI counts; WARNs on any unresolved POI
```

**Overpass-blocked?** A full rebuild needs the network, but a *POI-only* edit usually doesn't — so `build-map.py` has an **Overpass-free `--offline` mode** (also the automatic fallback when the fetch fails). It reconciles `POIS` against the *already-baked* `map-data.json`: alias changes flow straight through (pure metadata on POIs that already have geometry), and a new **address-anchored** POI is promoted from its baked building — every building carries its `a` (e.g. `"Liesing 20"`) + footprint `p`, so it becomes a pin (`xy` = centroid, `fp` = the footprint) and is dropped from `buildings`. So the flow is: edit `POIS`, run `python3 scripts/build-map.py --offline`, then bump `sw.js` `V` since `map-data.json` is precached. Only `osm`/`way`-anchored POIs genuinely need a real Overpass run (the mode WARNs and lists them).

`scripts/build-terrain.py` (uv + PEP 723: Pillow + numpy) bakes the two raster layers, reading the bbox **back from `map-data.json`'s `meta`** so they register pixel-for-pixel with the vector — so always run it *after* `build-map.py`:

```
python3 scripts/build-map.py        # vector + labels + meta.bbox
uv run scripts/build-terrain.py     # map-relief.jpg (Terrarium DEM hillshade) + map-aerial.jpg (basemap.at ortho)
```

**Not** in the pre-commit hook (network + slow; town geometry is ~static). `map-data.json` and both JPEGs are committed. **Attribution is required and per-layer** — `map.html` swaps it with the active base: Map → "© OpenStreetMap" (ODbL); Relief → "AWS Terrain Tiles"; Aerial → "© basemap.at · CC BY". Keep all three. (Esri/Google aerial imagery can't be bundled offline — basemap.at's CC BY orthophoto is why Aerial can be baked in.)

## Gotchas

- **gviz JSONP, not fetch** (CORS + view-only). If it ever returns blank: sheet sharing changed (must stay link-viewable) or that day's tab isn't posted — the app shows "not posted yet" rather than erroring.
- **iOS** has no background refresh; it updates on open — and since the home-screen card may *resume* (not reload), `app.js` re-pulls on `visibilitychange` and offers **pull-to-refresh** (`#ptr`). Cold start shows a "Loading…" state rather than falsely claiming "not posted." We chose **pure-pull on purpose** — push would re-add a scheduled sender, and home-screen web push is also DMA-restricted on EU-region devices.
- **Coda rotation** is deterministic by festival day (sourced tier first), so no state file is needed.
- **Bank integrity** — entries are tiered `sourced` / `attributed` / `standard`; never render a quote as more certain than its tier. Quotes are page-cited (Dvořák → Šourek; Bruch → Fifield). One landmine: the line *"I should be glad if something occurred to me…"* is **Brahms about Dvořák** (Šourek p. 15), not a Dvořák self-quote — it lives in the Brahms entry. Don't "fix" it back. When expanding the bank, prefer omitting a doubtful entry over including a spurious one.
- **`analytics.gs` edits don't deploy themselves** — the repo copy of `scripts/analytics.gs` is *reference only*; the live code is the Apps Script bound to Jason's private analytics sheet. Any change here must be hand-pasted into that script **and** re-deployed via **Deploy → Manage deployments → ✎ → Version: New → Deploy** (a plain save does NOT update the live `/exec` URL — Apps Script's classic trap). If a redeploy ever mints a *new* URL (e.g. "New deployment" instead of editing the existing one), `URL_` in `ping.js` must be updated + `sw.js` `V` bumped. Two invariants to keep in lockstep across `ping.js` ↔ `analytics.gs`: `TOK` (same literal in both), and the uid recipe (SHA-256 over UTF-8, first 4 bytes, lowercase hex — `uid()` in ping.js must equal `UID()` in the sheet, or the who→name key silently stops matching; sanity check: `UID("Jason Sundram") = "70f71792"`). When changing what a ping carries, remember old clients' queued pings flush with the *old* params — keep `doGet` tolerant of missing ones.
